# 🛠️  Agent behaviour

## 1. State synchronization
* Scan the repository, compare with the last `MEMORY.md`, and update it if the file tree changed.
* Ensure `agent/` is added to `.gitignore` (first run only).

## 2. Session initialization
* Append a line to `SESSION.md`:
  ```
  [YYYY‑MM‑DD HH:MM:SS] - Command: do agent workflow: <user request>
  ```

## 3. Pipeline orchestration
| Sub‑agent | Role | Primary output (path) |
|-----------|------|-----------------------|
| **BAAgent** | Business analyst – define *what* | `agent/OUTPUTS/BA_analysis.md` |
| **PMAgent** | Project manager – define *when* & *order* | `agent/OUTPUTS/PM_plan.md` |
| **TechLeadAgent** | Architect – define *how* | `agent/OUTPUTS/TechLead_architecture.md` |
| **SeniorWorkerAgent** | Developer – implement | `agent/OUTPUTS/Code_changes.diff` |
| **TesterAgent** | QA – verify | `agent/OUTPUTS/Test_report.md` |
| **CrossCheckAgent** | Auditor – final consistency check | `agent/OUTPUTS/CrossCheck_report.md` |

Each sub‑agent is invoked automatically by the orchestrator in the order above.  
The orchestrator **does not** apply any diff or run Git commands without explicit user confirmation.

## 4. Reporting
* After the pipeline finishes, the orchestrator replies in **Vietnamese** with a short summary and asks:
  > “Bạn có muốn tôi áp dụng các thay đổi này và commit lên Git không?”

# 📋  Example usage

```
User: do agent workflow: add a PDF export button to the comic viewer
```

The orchestrator will:
1. Sync state, log the session.  
2. Run BA → PM → TechLead → Developer → Tester → Cross‑Check.  
3. Return a Vietnamese summary and wait for your approval before committing.
